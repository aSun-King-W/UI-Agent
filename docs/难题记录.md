## 难题一：淘宝风控与反自动化对抗

### 现象

Playwright 操控的浏览器被淘宝风控系统识别为自动化工具，表现为：

1. **导航超时** — `goto` 使用 `waitUntil="networkidle"` 时，因淘宝持续有异步请求，30秒内永远无法达到"无网络请求"状态
2. **滑块验证码** — 搜索操作触发滑块验证，即使手动也无法在 Playwright Chromium 下滑动
3. **登录弹窗拦截** — 首页弹出登录浮层，但浮层中的`J_MIDDLEWARE_FRAME_WIDGET`遮挡层阻止点击操作
4. **指纹检测** — 淘宝检测到 `navigator.webdriver`、异常的 `plugins`、缺失的 `chrome.runtime` 等自动化特征

### 解决过程

#### 阶段一：修复导航策略

- **问题**: `waitUntil="networkidle"` 在淘宝页面上几乎永远无法满足
- **解决**: 将所有 `networkidle` 改为 `load`（等待初始 HTML+资源加载完成即返回），并将默认超时从 30s 提升到 60s
- **涉及文件**: `browser.py`, `login.py`, `search.py`, `filter.py`, `cart.py`, `page_utils.py`, `tools.py` 共7个文件

#### 阶段二：登录弹窗处理

- **问题**: 淘宝首页会弹出登录浮层，但原代码只处理导航到 `login.taobao.com` 独立页面的场景
- **解决**: 
  - 新增 `_try_popup_login()` 方法，检测首页浮层并直接在浮层中填写凭据
  - 新增 `_has_login_popup()` DOM 检测，让页面模式识别能感知登录弹窗
  - 双策略：弹窗优先 → 失败降级到独立登录页
- **涉及文件**: `login.py`, `page_utils.py`

#### 阶段三：Cookie 复用（最终方案）

- **问题**: 即使解决了导航和填写流程，淘宝的滑块验证码仍会阻止程序化登录
- **解决**: 
  - 新增 `manual_login.py` 脚本
  - 使用 `launch_persistent_context` 启动真实 Chrome（`channel="chrome"`），而非 Playwright 内置 Chromium
  - 用户手动登录一次 → Cookie 持久化 → 后续运行自动复用
- **关键**: 真实 Chrome 浏览器的指纹与正常用户一致，不会被风控拦截

### 经验总结

1. **不要对抗 `networkidle`** — 对于淘宝这类重度 SPA，`load` 比 `networkidle` 更实用
2. **Cookie 复用 > 自动登录** — 面对强风控平台，第一次手动登录保存 Cookie，后续直接加载是最稳的方案
3. **真实 Chrome > Playwright Chromium** — 淘宝的反自动化检测能识别 Playwright 内置浏览器的指纹，使用系统 Chrome 的 `launch_persistent_context` 能有效绕过
4. **遮挡层处理** — JS 直接 `display: none` 隐藏遮挡层比尝试点击关闭按钮更可靠
5. **多策略降级** — 任何时候都要有后备方案，弹窗不行就走独立页，自动不行就切手动

## 难题二：LLM Agent 编排的不确定性

### 现象

Agent 基于 LLM 决策下一步操作，但 LLM 的决策不稳定：

1. 导航失败后仍尝试搜索（不重试导航）
2. 登录弹窗出现时不调用 `login_taobao` 工具
3. 页面有遮挡层时不知道先移除遮挡层

### 应对策略

- **Prompt 优化**: 在 System Prompt 中明确典型流程和决策规则
- **页面模式感知**: 让 `get_page_state` 返回更准确的页面模式（如识别登录弹窗后返回 `"login"`）
- **工具健壮性**: 每个工具内部自包含容错逻辑（导航重试、遮挡层移除），不依赖 LLM 的"智能"
- **降级方案**: 核心的 Cookie 复用机制减少了 LLM 需要做决策的环节（跳过登录步骤）

### 思考

LLM Agent 编排适合"正常情况下的灵活调度"，但在异常恢复场景下表现不稳定。
对于生产级 UI 自动化，建议关键路径（登录、搜索）走确定性代码，
只有弹性环节（如页面结构变化时的自适应定位）才交给 LLM。

## 难题三：淘宝登录页结构变化

### 现象

淘宝登录页频繁改版，CSS 选择器（`#fm-login-id`、`.password-login` 等）可能失效。

### 应对

- 使用多选择器 fallback（每个操作备选 3-5 个选择器）
- 最后兜底：找到页面上任意可见的 `<input>` 和 `button`
- LLM 辅助：通过 `get_page_state` 让 Agent 阅读页面文本内容来自适应定位

---

## 遗留问题

1. **搜索滑块验证** — 搜索操作仍可能触发滑块，目前靠手动滑动解决。后续可考虑加入滑块检测 + 等待机制
2. **Cookie 有效期** — 淘宝 Cookie 有效期不确定，过期后需要重新运行 `manual_login.py`
3. **飞书长连接稳定性** — `KeyboardInterrupt` 中断显示飞书 WebSocket 连接在 Windows 上可能被系统事件中断
4. **搜索框点击** — 淘宝首页的 `J_MIDDLEWARE_FRAME_WIDGET` 遮挡层在不同页面版本下可能需要更强力的移除策略
